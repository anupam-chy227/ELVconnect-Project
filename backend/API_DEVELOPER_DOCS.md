# ELV CONNECT - Developer API Documentation (v1.2)

Welcome to the ELV CONNECT API documentation. This guide is designed to help developers integrate with our platform efficiently, focusing on robust error handling and complex business workflows.

## 🚀 1. Getting Started

### Base URL
- **Local**: `http://localhost:5000/api/v1`
- **Environment Variable**: `{{baseUrl}}`

### Authentication
All protected endpoints require a Bearer Token in the header:
`Authorization: Bearer <your_jwt_token>`

### Request Tracking (Best Practice)
To help with debugging, we recommend sending a `X-Correlation-ID` in your headers. This ID will be returned in the response and can be used to trace requests in logs.

## 🛠 2. Standardized Error Handling (RFC 9457)
All error responses follow this machine-readable structure:
```json
{
  "type": "https://api.elvconnect.com/errors/validation-failed",
  "status": 422,
  "title": "Unprocessable Entity",
  "detail": "The budget amount cannot be less than zero.",
  "instance": "/api/v1/jobs",
  "requestId": "req-123456789"
}
```

### Common Status Codes
| Code | Meaning | Scenario |
|---|---|---|
| **200/201** | Success | Request completed successfully. |
| **400** | Bad Request | Malformed JSON or invalid syntax. |
| **401** | Unauthorized | Missing or expired JWT token. |
| **403** | Forbidden | You are logged in but don't have permission (e.g. Service Provider trying to post a job). |
| **404** | Not Found | Resource ID does not exist. |
| **422** | Semantic Error | Validation passed but logic failed (e.g. assigning a job that is already completed). |
| **429** | Too Many Requests | Rate limit exceeded. Please retry after 60s. |

## 💼 3. Job Lifecycle & Workflow Scenarios

### **Scenario: Post a New Job**
- **Endpoint**: `POST /jobs`
- **Status**: Transitions to `open`.
- **Error Possibility**: `400` if required fields like `location` are missing.

### **Scenario: Approve/Accept an Application**
- **Endpoint**: `POST /jobs/:id/applications/:appId/approve`
- **Logic**: Changes application status to `accepted` and Job status to `in_progress`.
- **Error Possibility**: `422` if the job is already `cancelled` or `assignedTo` someone else.

### **Scenario: Discard/Cancel a Job**
- **Endpoint**: `DELETE /jobs/:id` or `PATCH /jobs/:id/status`
- **Logic**: Soft-deletes the job or sets status to `cancelled`.
- **Error Possibility**: `403` if the user is not the owner (Customer) of the job.

## 🧾 4. Invoice & Billing Scenarios

### **Scenario: Approve & Pay Invoice**
- **Endpoint**: `POST /invoices/:id/payments`
- **Logic**: Transitions status from `sent` -> `partially_paid` -> `paid`.
- **Error Possibility**: `422` if the payment amount exceeds the `balanceDue`.

### **Scenario: Change/Modify a Sent Invoice**
- **Endpoint**: `PATCH /invoices/:id`
- **Logic**: Reverts status to `draft` if significant changes are made, or keeps it `sent`.
- **Error Possibility**: `403` if the invoice is already marked as `paid` (modifications locked).

### **Scenario: Discard/Cancel Invoice**
- **Endpoint**: `PATCH /invoices/:id/status`
- **Status**: `cancelled`.
- **Error Possibility**: `404` if the invoice ID is incorrect.

## 🚨 5. Security & Rate Limiting
- **Internal Details**: We never return stack traces in production.
- **Rate Limit**: Max 100 requests per 15 minutes for standard users.
- **Data Privacy**: Personal identifying information (PII) is masked in logs.
